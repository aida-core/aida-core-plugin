# Security Policy

## Supported Versions

| Version | Supported              |
| ------- | ---------------------- |
| 1.5.x   | :white_check_mark:     |
| < 1.5   | :x: (please upgrade)   |

Only the latest 1.x release line receives security fixes. Once 2.x
ships, the supported window will be the latest 1.x **and** the
latest 2.x for a transition period.

## Reporting a Vulnerability

**Please do not open public issues for security vulnerabilities.**

### Preferred: GitHub Security Advisories

Report through
[GitHub Security Advisories](https://github.com/aida-core/aida-core-plugin/security/advisories/new).
That lets us discuss and fix the issue privately before public
disclosure.

### Alternative: Email

If you can't use Security Advisories, send an email to
<github@oakensoul.com> with the subject line
`[SECURITY] aida-core — <brief description>`.

### What to Include

- A description of the vulnerability and its potential impact
- Steps to reproduce or a proof-of-concept
- The version(s) of aida-core-plugin affected
- Any suggested fix, if you have one

### Response Timeline

- **Acknowledgement** — within **72 hours** of your report
- **Initial assessment** — within 7 days
- **Fix or mitigation** — as soon as reasonably possible; we'll
  coordinate disclosure timing with you

## Scope

This policy covers:

- The **`aida-core-plugin`** package itself (skills, agents,
  scripts, templates shipped from this repo)
- **Direct dependencies** in `requirements.txt` / `requirements-dev.txt`
  that ship as part of the plugin install
- **Scaffolded plugin templates** — security-relevant defaults in
  the files generated by `/aida plugin scaffold` (e.g., CI workflow
  hardening, lint configurations)

This policy does **not** cover:

- Plugins authored by third parties using the AIDA scaffolding
- Downstream content stored in `.claude/` directories
- Issues in Claude Code itself (please report those to Anthropic)
- Issues in third-party MCP servers configured by AIDA users

## Public Acknowledgement

Reporters of confirmed vulnerabilities are credited in the
release notes for the fix, unless they prefer to remain anonymous.
